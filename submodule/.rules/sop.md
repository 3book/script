
## IPD (Integrated Product Development)

![workflow.svg](assets/ipdFlow.png)

### 各阶段目标

| 节点 | 阶段       | 目标                                                   |
| ---- | ---------- | ------------------------------------------------------ |
| TR1  | 概念       | 确认需求、概念设计、原型验证、关键技术识别、可行性分析 |
| TR2  | 计划       | 明确产品规格、系统架构、关键技术路径                   |
| TR3  | 需求分析   | 明确逻辑需求，完成概要设计、模块划分、关键接口设计     |
| TR4  | 详细设计   | 完成详细设计、关键测试                                 |
| TR4A | 集成测试   | 完成软硬件集成测试                                            |
| TR4B | 样机验证   | 完成工程样机测试                                       |
| TR4C | 客户跨验   | 完成客户端跨机测试                                     |
| TR5  | 小批量试产 | 小批量生产、环境/可靠性验证、试用反馈                  |
| TR6  | 发布量产   | 确认可量产性、工艺稳定、质量体系、发布准备             |

## 逻辑开发SOP

![workflow.svg](assets/workFlow.svg)

### 逻辑开发关键活动说明

### 项目计划必须包含

- 项目目标
- 制定关键里程碑
- 制定人员计划
- 确定开发工具链
- 评估开发活动所需设备、耗材、单板
- 配置管理（项目仓库、QA）
- 识别人员、技术风险
- 流程遵从说明
- 模块划分和职责

### 包需求分解必须

- 分解为 fpga 的模块需求、接口需求
- 分解后的需求必须注明需求来源
  - 客户
  - 标准规范
  - 系统需求

### 需求分析必须分析

- 功能
  - 阐明功能特性
  - 阐明规格要求
  - 评估 serdes、PLL、RAM、LUT 等资源要求
- 接口
  - 明确执行标准、协议
  - 明确信号定义、位宽
  - 明确接口速率
- 性能
  - 明确吞吐要求
  - 明确延迟要求
- DFx
  - 可维测需求
  - 可靠性需求
  - 可扩展性需求

### 概要设计必须

- 绘制总体框图
- 说明数据流向
- 设计关键模块方案
- 明确对外接口
- 明确关键数据格式
- 明确对外接口协议

### 概要设计必须阐明
- 技术风险
- 外部依赖
- 应对措施

### 初始pin工程必须

- 必须包含所有外部接口
- 编译通过
- 确认所需功能未被优化

### TR3 评审

- 对象 《逻辑需求分析文档》
- 时间 TR3 且需求分析文档初稿完成后
- 人员 逻辑项目全员、上下游相关人员、逻辑专家、系统工程师
- 输出 评审纪要、评审通过后的《逻辑需求分析文档》

### 需求变更必须

- 提交申请
- 分析影响
- 更新《逻辑需求分析文档》
- 经系统组评审

### 详细设计必须

- 绘制模块框图
- 说明数据处理过程
- 罗列接口定义
- 关键数据格式
- 接口时序图
- 资源估计
- 在硬件原理图设计中期对硬件原理图管脚进行工程编译

### 编译必须

- 基于 dev 分支 pull
- 使用一致工具链
- 分析综合、资源、时序报告
- 分析告警、违例，并给出例外说明

### 自验必须

- 执行功能遍历测试
- 针对性测试修改部分

### 分支策略

#### main

- 该分支仅支持 merge 操作，禁止提交，已做限制
- 对完发布版本仅从该分支
- 单项目需对外发布不同特性 main 不唯一

#### dev

- 该分支提供完整的开发版本编译
- feature 分支 merge，然后自测，方能 push

#### feature

- 该分支为功能开发分支
- 功能特性开发者自行创建
- 开发自验完成后 merge 到 dev 分支
